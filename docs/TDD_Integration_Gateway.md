## 1.	Цели
  a.	**Бизнес-задача:** Обмен данными между [Система 1] и [Система 2] осуществляется через ввод в UI [Система 2] сотрудниками отдела НСИ. Это приводит к задержкам до 4 часов (среднее время реагирования специалиста), риску возникновения ошибок из-за «человеческого фактора» и искажению данных при передаче.
  
  b.	**Решение:** разработать интеграционный сервис (Integration Gateway), обеспечивающий автоматическую передачу и трансформацию данных между [Система 1] и [Система 2] с поддержкой XML и JSON.
  
  c.	**Ценность:** Сокращение времени передачи данных с нескольких часов до секунд, снижение операционных расходов за счёт автоматизации и снижение количества ошибок ручного ввода.

## 2.	Границы
  a.	**In-Scope (Входит в текущую итерацию)**:<br>
   &nbsp; i.	Создание Integration Gateway для передачи данных между [Система 1] и [Система 2].<br>
    &nbsp; ii.	Реализация Mapping Engine для трансформации XML в JSON и наоборот.<br>
     &nbsp; iii.	Настройка Retry Policy (экспоненциальная задержка) для обработки сбоев сети.<br>
      &nbsp; iv.	Валидация данных на соответствие XSD-схемам.<br>
    
  b.	**Out-of-Scope (Не входит в текущую итерацию)**:<br>
   &nbsp; i.	UI-интерфейс: В текущей итерации работа только через API.<br>
    &nbsp; ii.	Масштабирование: Нагрузочное тестирование на >1000 запросов в секунду.<br>
     &nbsp;  iii.	Аналитика: Построение графиков и отчетов по работе системы.<br>
      &nbsp;  iv.	Архивация: Перенос старых маппингов в долгосрочное хранилище.<br>
       &nbsp;   v.	Обработка файлов: Функционал передачи файлов исключен из текущей итерации, так как входящие XML-структуры не содержат бинарных данных.<br>
        &nbsp;   vi.	API администрирования: Реализация API-интерфейса для управления реестром правил (CRUD) вынесена в следующую итерацию. В рамках текущего проекта управление правилами (registry_rules) осуществляется администраторами БД посредством SQL-запросов.<br>

b.	**Описание связей**:<br>
&nbsp; i.	Между таблицами `registry_rules` и `rule_storage` установлена связь типа 1:1 (один-к-одному). <br>
 &nbsp; ii.	Поле `rule_guid` в таблице `registry_rules` является внешним ключом, ссылающимся на первичный ключ `rule_guid` в таблице `rule_storage`. <br>
  &nbsp; iii.	Данная структура обеспечивает высокую скорость работы, «тяжелый» контент (JSON-тело правила) хранится в отдельной таблице для оптимизации производительности запросов метаданных. <br>
   &nbsp; iv.	Реализовано каскадное удаление (ON DELETE CASCADE): при удалении записи из таблицы `registry_rules` соответствующая запись в `rule_storage` удаляется автоматически. <br>
   
## 5.	Правила трансформации данных
a.	**Иерархия маппинга:** <br>
&nbsp;i.	Общие правила <br>
&nbsp; &nbsp; 1.	Обработка null: Все пустые или отсутствующие в XML теги при конвертации в JSON должны превращаться в null. Это необходимо для корректной валидации полученных данных в [Система 2]. <br>
&nbsp; &nbsp; 2.	Формат времени: Вся работа со временем ведется исключительно в UTC. Любое поле типа `dateTime` конвертируется в строку вида `2026-03-12T14:30:00Z`. Использование локальных форматов (типа DD.MM.YYYY) в рамках интеграционного слоя запрещено. <br>
&nbsp; &nbsp; 3.	Именование полей: При трансформации в JSON все имена тегов переводятся в `snake_case`. Если в XML тег называется `<OrderDate>`, в JSON следует указать название как `order_date`. <br>

&nbsp; &nbsp; *4.	Таблица 4. Соответствие форматов* <br>
| **Тип в XML (XSD)** | **Тип в JSON** | **Правило трансформации**               |
|---------------------|----------------|------------------------------------------|
| xs:string           | string         | Прямое копирование (UTF-8)               |
| xs:integer          | number         | Приведение к целочисленному типу         |
| xs:decimal          | number         | Приведение к числу, сохранение точности  |
| xs:boolean          | boolean        | true/1 -> true, false/0 -> false         |
| xs:dateTime         | string         | ISO 8601 (YYYY-MM-DDThh:mm:ssZ)          |
| xs:date             | string         | ISO 8601 (YYYY-MM-DD)                    |
| xs:anyURI           | string         | Передача URL как строки                  |

&nbsp; ii.	Специальные правила:<br>
&nbsp;&nbsp; *Таблица 5. Трансформация комплексных элементов.* <br>
| **Тип в XML (XSD)** | **Тип в JSON** | **Правило трансформации**                   |
|---------------------|----------------|----------------------------------------------|
| xs:complexType      | object         | Вложенный JSON-объект                        |
| xs:sequence         | array          | Конвертация повторяющихся тегов в массив []  |

## 6.	Отказоустойчивость

a.	**Политика повторных попыток (Retry Policy)** <br>
При возникновении сбоев при передаче данных (ошибки сетевого уровня или недоступность [Систем 1/2]), сервис применяет механизм повторных попыток с экспоненциальной задержкой. Количество попыток — 4 (всего 5 обращений, включая первую попытку). Интервалы: 1с, 2с, 4с, 8с. <br>
После 4-й неудачной попытки событие перемещается в Dead Letter Queue (DLQ) для последующего ручного анализа или оповещения администратора.<br>

b.	**Механизм транзакционного отката (Rollback).** <br>
Для обеспечения целостности данных при сбое на этапе трансформации или отправки в каждую из принимающих систем сервис использует механизм транзакционного управления: <br>
&nbsp; •	Принцип атомарности: Процесс трансформации рассматривается как единая транзакция. Если на любом этапе (валидация, трансформация, отправка) возникает критическая ошибка, система выполняет откат изменений.<br>
&nbsp; •	Поведение при ошибке: <br>
 &nbsp; &nbsp; 1.	Статус записи в таблицах `rule_storage` или `registry_rules` не обновляется (остается в исходном состоянии). <br>
 &nbsp; &nbsp; 2.	Сообщение не считается «обработанным» до получения подтверждения от принимающей стороны. <br>
 &nbsp; &nbsp; 3.	Логи с ошибкой записываются в систему мониторинга для последующего анализа.<br>

c.	**Сервис возвращает следующие стандартные коды состояний HTTP для обеспечения единообразия интеграции:** <br>

*Таблица 6. Коды ошибок.* <br>
 **Код ошибки**            | **Описание**         | **Причина возникновения**                                        |
|---------------------------|----------------------|-------------------------------------------------------------------|
| 200 OK                    | Успех                | Трансформация выполнена, данные доставлены                        |
| 400 Bad Request           | Ошибка валидации     | Некорректная структура XML или несоответствие XSD                 |
| 404 Not Found             | Ресурс не найден     | Не найден `mapp_id` или правило трансформации для указанной версии  |
| 422 Unprocessable Entity  | Ошибка трансформации | Ошибка при применении правил                                      |
| 500 Internal Server Error | Критическая ошибка   | Ошибка сервера или базы данных `rule_storage`                     |
| 503 Service Unavailable   | Ошибка связи         | [Система 1/2] недоступна (после исчерпания лимита Retry).         |
